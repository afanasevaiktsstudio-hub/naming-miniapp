# Низкоуровневый план: Telegram-бот для нейминга и брендинг-картинок

## Цель MVP
Собрать Telegram-бота, который:
- принимает английское название;
- возвращает 5 русифицированных вариантов;
- позволяет выбрать вариант кнопкой;
- генерирует брендинг-картинку по выбранному варианту.

## Формат совместной работы
На каждом шаге фиксируем:
1. Что делаем.
2. Что должно получиться (артефакт).
3. Как проверяем.
4. Условие перехода к следующему шагу.

## Шаг 0. Создаем и регистрируем Telegram-бота
**Делаем**
- Открываем `@BotFather`.
- Выполняем `/newbot`, задаем имя и username.
- Получаем `BOT_TOKEN`.
- Опционально настраиваем `/setdescription`, `/setabouttext`, `/setuserpic`.

**Артефакт**
- Зарегистрированный бот и валидный `BOT_TOKEN`.

**Проверка**
- Запрос `https://api.telegram.org/bot<BOT_TOKEN>/getMe` возвращает `ok: true`.

**Готово, если**
- Токен сохранен безопасно и бот доступен в Telegram.

## Шаг 1. Подтверждаем технические решения
**Делаем**
- Фиксируем стек: Python 3.11+ и `aiogram`.
- Фиксируем API-провайдеров (текст и картинки).
- Фиксируем формат ответа нейминга: строгий JSON.

**Артефакт**
- Короткая спецификация MVP в `README` или отдельном файле.

**Проверка**
- Согласованы поля JSON и UX-кнопки.

**Готово, если**
- Нет открытых вопросов по стеку и API.

## Шаг 2. Базовый каркас проекта
**Делаем**
- Создаем структуру:
  - `app/handlers`
  - `app/services`
  - `app/prompts`
  - `app/keyboards`
  - `app/storage`
- Добавляем `requirements.txt`.
- Добавляем `app/main.py`, `app/config.py`, `.env.example`.

**Артефакт**
- Проект запускается без бизнес-логики.

**Проверка**
- Приложение стартует без импорт-ошибок.

**Готово, если**
- Бот отвечает на `/start`.

## Шаг 3. Конфиг и безопасный запуск
**Делаем**
- Подключаем `BOT_TOKEN`, `OPENAI_API_KEY`, модели текста и картинок из `.env`.
- Добавляем валидацию обязательных переменных при старте.
- Включаем базовое логирование.

**Артефакт**
- Предсказуемый старт с понятными ошибками конфигурации.

**Проверка**
- При пустом `.env` сервис падает с понятной ошибкой.
- При валидном `.env` сервис запускается.

**Готово, если**
- Конфиг централизован и не дублируется по модулям.

## Шаг 4. Входящий текст -> 5 вариантов названия
**Делаем**
- Реализуем `NamingService`.
- Создаем системный промпт в `app/prompts/naming_prompt.py`.
- Требуем JSON-ответ формата:
  - `{"variants":[{"title":"","style":"","comment":""}]}`
- Добавляем защиту: retry при невалидном JSON.

**Артефакт**
- На ввод `Upside Towers` возвращается массив из 5 вариантов.

**Проверка**
- В ручном тесте бот стабильно возвращает 5 читаемых RU-вариантов.

**Готово, если**
- На 10 тестовых запросах нет падений парсера.

## Шаг 5. UX выбора через inline-кнопки
**Делаем**
- Показываем варианты и кнопки выбора.
- Сохраняем выбранный вариант в `SessionStorage` (in-memory для MVP).
- Добавляем кнопку `Еще варианты`.

**Артефакт**
- Пользователь выбирает конкретный вариант одним кликом.

**Проверка**
- После выбора бот подтверждает выбранное имя и показывает следующий action.

**Готово, если**
- Все callback-сценарии работают без ошибок.

## Шаг 6. Генерация брендинг-картинки
**Делаем**
- Реализуем `ImageService` и шаблон промпта под брендинг.
- Добавляем кнопку `Сгенерировать картинку`.
- Отправляем картинку в Telegram как фото.

**Артефакт**
- По выбранному названию приходит 1 изображение.

**Проверка**
- На 3-5 разных названиях генерация отрабатывает без падений.

**Готово, если**
- Ошибки image API обрабатываются и показывают понятный текст пользователю.

## Шаг 7. Минимальная надежность
**Делаем**
- Добавляем rate limit на пользователя.
- Добавляем валидацию входа (пустой текст, длина).
- Добавляем централизованный error handler.

**Артефакт**
- Бот устойчив к спаму и неверному вводу.

**Проверка**
- Лимиты реально срабатывают.
- Необработанные исключения отсутствуют в основном сценарии.

**Готово, если**
- Бот не падает при типовых ошибках внешнего API и пользовательского ввода.

## Шаг 8. Тесты и smoke-check
**Делаем**
- Unit-тесты: парсинг LLM JSON и построение клавиатур.
- Smoke-checklist ручного сценария.

**Артефакт**
- Минимальный тестовый контур + чеклист проверки.

**Проверка (smoke)**
- `/start` работает.
- Текст -> 5 вариантов.
- Выбор варианта -> кнопка генерации.
- Генерация картинки -> фото отправлено.

**Готово, если**
- Весь smoke-checklist проходит без критических сбоев.

## Шаг 9. Деплой MVP
**Делаем**
- Добавляем `Dockerfile`.
- Деплоим на Railway/Render/Fly.
- Настраиваем переменные окружения.
- Стартуем с long polling (потом можно перейти на webhook).

**Артефакт**
- Публично доступный бот в Telegram.

**Проверка**
- End-to-end сценарий проходит в прод-окружении.

**Готово, если**
- Бот стабильно отвечает и генерирует картинки в проде.

## Definition of Done для MVP
- Реализован end-to-end поток: ввод -> варианты -> выбор -> картинка.
- Ошибки API не роняют бота и отображаются пользователю.
- Есть базовые лимиты и логирование.
- Бот задеплоен и доступен внешним пользователям.

## Оценка сроков
- Шаги 0-3: 2-4 часа.
- Шаги 4-6: 6-10 часов.
- Шаги 7-9: 4-8 часов.

Итого: 2-3 дня фокусной работы на качественный MVP.
